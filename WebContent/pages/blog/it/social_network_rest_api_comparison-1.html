After having spent one year in designing and developing a membership solution based on Gigya RaaS for the Canadian media giant Bell Media, I was curious to understand better the effort in consuming web api for main social networks, which have been collected and abstracted from Gigya in my task. Bell company owns many media websites (dozens) which used a personal membership flow, different per each site, with a final result composed of redundancy on frontend/backend side, slow maintenance process and difficulties in sharing users between sites. For this reason, in according with customer, we realized a solution starting by the idea to have only a single membership site, pointed from all the others but still able to guarantee custom functionalities where required.

Let's imagine for example a site where editors allow people to do the login by twitter and facebook, then another site of the same company where editors prefer linkedin rather than google+ and so on, well this scenario means to develop an out-and-out service to configure and consume backend web api for all required social networks, including all interaction flows as registration, authentication, reset password et cetera et cetera. That is what Gigya does and what I was curious to estimate in terms of time and complexity.

Well done Gigya but now it's my turn. To "play" with that I defined 4 simple rules:

  1. only backend Rest API
  2. get user info and the main feature for the specific social
  3. 3 hours as max developing time for a first draft
  4. re-use unofficial libraries if exceed 3 hours

Following the list of Rest API I tested:

Spotify

I start by saying Spotify and Instagram share the top for different aspects and this means I found Spotify Rest API so simple, smart and rich of details. In 3 hours I was able to get back user info and build the playlist list including an iframe player per each playlist.

Playlist with player example

PROS
Perfect documentation, amazing samples for many languages, easy to consume.
CONS
I didn't find a way to set a picture for my user.

We are talking about REST Api designed with OAuth 2.0 as authentication model and json as response format. My personal website www.danieleautizi.com is developed in java, for this reason I used the example project Spotify Rest API Java to start with.

Test live my Spotify implementation

Instagram

My favourite social network ever? Instagram! The documentation is super complete even if the samples are not as much as I expected, they follow the standard RestFul design and in adding the OAuth 2.0 as authentication protocol. I was spending more than 2 hours to handle the authentication in a fancy way, for this reason I used the library jInstagram. Getting user info and last 20 pictures have been posted for the specific account was just a child's play.

Instagram REST Api can support json and jsonp as response format. I noticed high performance in response even with quite complex requests.

PROS
Amazing documentation, pretty easy to consume, interesting features as geo-localization.
CONS
Not enough samples for different languages.

Test live my Instagram implementation

Linkedin

Lately Linkedin represents the most efficient "tool" in recruitment of qualified resources. Actually I think it comes from a great idea and it works well for the original purpose. I spent only 1 hour to realize I would not get more info than I had already got. The free license key does not allow to have previous positions and technologies tags that represent an important feature in my opinion.

User work position example

PROS
Interesting scenario to make business in recruitment sector, documentation good enough.
CONS
Lack of functionalities with the free account.

Again simple REST Api designed with OAuth 2.0 as authentication model but they provide xml in adding to json and jsonp as response format.

Test live my Linkedin implementation

Google+

This Google's "baby" is probably the less used product its team has ever created but I was too much curious to test it. Consuming Google+ Rest API is not distant from consuming any other google service, good documentation, smart and clean sample code. I had some problems in setting the callback for local environment, even if it was well configured on google application panel. Anyway 3 hours are enough if I don't consider what I wasted to solve the callback bug.

I think sometimes having many different solutions to the same problem, adds complexity and doubts. However no doubt on google code quality.

PROS
Easy to use samples, full set functionalities, high performance, many solutions in consuming ws.
CONS
Configuration is less intuitive than the others.
